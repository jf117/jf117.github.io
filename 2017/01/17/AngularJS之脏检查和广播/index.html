AngularJS之脏检查和广播

发表于 2017-01-17 | 分类于 AngularJS

MVVM

  • AngularJS 采用的就是 MVVM 这种视图数据分离的架构
  • 此外还有 MVC 和 MVP 两种架构方式
  • 参见阮一峰博客

双向数据绑定

  • 监听js里面值的变化：写一个循环执行的代码取一段时间去取一下值，看看变了没有
  • 监听到当视图层的数据发生变化就会将数据返回给数据层
  • 这样就形成了数据的双向绑定
  • 虽然 AngularJS 最突出的特点就是数据的双向绑定，但是使用最多的还是从数据层到视图层的单向绑定

$watch

那又是如何实现监听js里面的数据变化的呢？

使用 $watch 监听 $scope 里面值的变化。

AngularJS 是如何知道什么时候触发 $watch？又如何取触发？

  • 如果你是作者怎么办
    • 找一个函数来循环去检查scope中的数据是否发生了变化，
    • 发现有变化去更改相应的值
  • 这个函数就是 —->$scope.$digest()

  • 当更改了$scope里面的值、$timeout、$interval等，自动触发一次$digest循环

  • 但是如果使用 $apply() 这个方法，就会告诉 angular 进行数据更新

$digest

  • $digest 循环最少也会运行两次，即使在 $watch 的回调函数中并没有改变任何 model 。
  • 如果 $watch 的回调函数中修改了 $scope，又会调用 $watch 事件，这样就会进入死循环。
  • 为此 angularjs 设置了这样一个机制，$digest() 最多迭代10次

$apply()

  • $apply() 实现脏检查实际上是通过$digest实现的。
  • $digest 是一个内部函数，我们没法调用。
  • 即使是 AngularJS 也并不直接调用 $digest()。
  • 原因是在 $apply() 与 $digest() 之间还有一层很重要的 $evel() 判断。如果中间有错误 $evel() 将会把错误交送到 $exceptionHandler 进行处理。
  • 不直接调用 $digest()，而是调用 $scope.$apply()，然后 $scope.$apply() 调用 $rootScope.$digest()。

脏检查

  • 脏检查的检查范围
    • 不会脏检查所有的对象。当对象被绑定到 html 中后，这个对象才会添加为检查对象（watcher）。
    • 不会脏检查所有的属性。同样当属性被绑定后，这个属性才会被列为检查的属性。
    • 在 angular 程序初始化时，会将绑定的对象的属性添加为监听对象（watcher），也就是说一个对象绑定了 N 个属性，就会添加 N 个 watcher。
  • angular 什么时候去脏检查呢？
    • angular 所系统的方法中都会触发比较事件，比如：controller 初始化的时候，所有以 ng- 开头的事件爱你执行后，都会出发脏检查。
    • 必要的时候我们要手动的触发脏检查：$apply() 仅仅只是进入 angular context，然后通过 $digest() 触发脏检查。
      • $apply() 如果不给参数的话，会检查该 $scope 里的所有监听的属性，推荐给上参数。

什么时候需要我们手动调用 $apply() 方法？

不在 AngularJS 的控制范围内去修改$scope里的值
例如setInterval

$digest() 的遍历方式

  • 遍历的方式有深度遍历和广度遍历之分。
  • AngularJS的遍历是从 $rootScope 开始遍历(深度优先)检查数据变更。

广播和发射

  • 通过 $broadcast() 给所有子级广播一个自定义的事件名称和指定的内容
    • 只有监听这个事件名称的子级会接收到父级广播的内容
  • 通过 $emit() 给父级发射一个自定义的事件名称和指定的内容
    • 只有监听这个事件名称的父级会接收到子级发射的内容
